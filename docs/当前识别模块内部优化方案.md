# 当前识别模块内部优化方案

版本：V1.0  
适用范围：考试文件识别、答题卡模板切分、学生答题卡切分和身份区域隔离模块  
方案目标：在不影响现有对接接口的前提下，借鉴现有阅卷系统的有效业务能力，完善当前识别模块的内部管理、复核和结果查询能力。

## 一、方案结论

当前模块不直接引入参考项目的 Django、MySQL、Redis 或评分服务，也不替换现有 OCR 和切分算法。

本次优化采用“外部接口不变、内部能力增强、存储可替换”的方式：

```text
现有输入接口
    -> 当前文件解析和 OCR
    -> 当前自由版式识别 / 固定模板切分
    -> 当前身份区域隔离和脱敏
    -> 新增内部任务记录、复核记录和结果查询
    -> 现有 exam-document.v1 输出
```

核心原则：

1. 成功请求的输入字段不变。
2. `exam-document.v1` 输出结构不变。
3. `question_id`、`exam_id`、`submission_id` 和模板版本继续作为核心关联字段。
4. 姓名、学号、专业、学院、年级等身份信息继续只保存在受保护的内部记录中。
5. 身份信息不得进入公共 `questions[]`、`answers[]` 或批改模块输入。
6. 识别模块只负责文件处理和结果输出，不负责评分和评分建议。
7. 新增能力优先通过内部服务和查询接口提供，不把内部字段直接塞进原有输出。

## 二、当前基础和现存不足

### 2.1 当前已经具备的能力

当前模块已经具备以下基础：

- 支持 PDF、DOCX、JPG、PNG 等文件解析；
- 支持自由版式试题卷识别；
- 支持空白答题卡模板生成；
- 支持教师复核答题区域；
- 支持固定模板答题卡切分；
- 支持姓名、学号、专业、学院、年级身份区域；
- 支持身份区域与答题区域分离；
- 支持统一的 `question_id`；
- 支持考试与模板绑定校验；
- 支持结果持久化和重复提交缓存；
- 支持置信度和风险标记；
- 支持脱敏后的公共输出；
- 已有 `exam-document.v1` Schema 和接口文档。

### 2.2 当前需要补充的内部能力

当前项目仍然偏向“单次文件处理模块”，还缺少参考阅卷系统中的部分管理能力：

- 无统一的识别任务记录；
- 前端无法稳定查询某次处理的状态；
- 复核结果、处理结果和原始文件之间的关系不够集中；
- 结果查询主要依赖处理文件或内部存储；
- 内部数据记录和未来数据库之间没有清晰的适配边界；
- 批量处理时缺少统一的成功、失败、待复核统计；
- 错误、复核和结果状态没有形成完整闭环。

这些问题不需要通过改动公共输出结构解决，可以在模块内部增加管理层。

## 三、借鉴参考阅卷系统的范围

### 3.1 借鉴内容

参考项目中以下能力值得借鉴：

- 考试、题目、学生提交和单题答案之间的关联方式；
- 学生答卷的处理状态；
- 单题图片和 OCR 文本同时保存；
- 教师可以修改 OCR 文本并确认；
- 批量处理任务有进度、成功数和失败数；
- 处理结果可供后续阅卷模块查询。

### 3.2 不直接借鉴的内容

以下内容不直接复制到当前识别模块：

- 参考项目每两页固定组成一份试卷的规则；
- 只取页面最大矩形作为答题区域的规则；
- 依赖图片文件名推断题号的规则；
- 将学生姓名和学号直接放入批改数据的方式；
- 将百度 OCR 等外部服务用于处理学生原始答卷的方式；
- AI 评分、评分理由和最终分数逻辑；
- 参考项目自己的数据库表结构和接口格式。

## 四、优化后的内部架构

建议在当前模块内部形成以下六层：

```text
1. 文件接收层
2. 识别处理层
3. 模板和考试注册层
4. 身份隔离与脱敏层
5. 任务与复核管理层
6. 对外结果适配层
```

### 4.1 文件接收层

职责：

- 校验文件扩展名和 MIME 类型；
- 生成稳定的 `submission_id` 和内部 `file_id`；
- 保存原始文件、文件大小和 SHA-256；
- 防止同一个提交被重复处理；
- 返回通俗中文错误信息。

不得改变现有调用方传入方式。现有 `file`、`document_role`、`exam_id`、`submission_id` 等字段继续使用。

### 4.2 识别处理层

保持现有逻辑：

```text
文档解析
  -> 页面渲染
  -> 图像预处理
  -> PaddleOCR
  -> 自由版式题目切分或固定模板区域切分
  -> 结果导出
```

自由版式试题卷：

- 识别题号和题目文本；
- 合并跨页题目；
- 输出 `questions[]`；
- 低置信度结果标记为需要复核。

固定模板答题卡：

- 通过 `exam_id` 服务端解析已绑定模板；
- 对扫描页进行页面对齐；
- 只裁剪模板中配置的答题区域；
- 单独裁剪身份区域；
- 输出 `answers[]`；
- 身份字段不进入答案结果。

### 4.3 模板和考试注册层

继续使用当前的考试和模板注册逻辑，但明确划分三类记录：

#### 考试记录

```json
{
  "exam_id": "mayuan_1516",
  "exam_name": "马克思主义原理15-16期末考试",
  "status": "draft",
  "question_catalog_version": 1,
  "template_id": "tpl_mayuan_1516_v1",
  "template_version": 1
}
```

#### 模板记录

保存：

- 模板 ID；
- 模板名称；
- 模板版本；
- 发布状态；
- 页面尺寸；
- 答题区域；
- 身份区域；
- 区域复核状态；
- 模板创建来源。

#### 提交记录

保存：

- `submission_id`；
- `exam_id`；
- 使用的模板 ID；
- 使用的模板版本；
- 原始文件哈希；
- 处理状态；
- 处理结果位置；
- 身份信息内部记录位置；
- 创建和更新时间。

考试、模板和提交之间的关系必须是：

```text
一个考试 -> 一个明确绑定的已发布模板版本
一次答题卡提交 -> 一个考试和一个模板版本
一次提交 -> 一个稳定的 submission_id
```

### 4.4 身份隔离与脱敏层

处理顺序必须保持为：

```text
页面对齐
  -> 身份区域裁剪
  -> 答题区域裁剪
  -> 身份区域单独 OCR
  -> 身份信息写入内部保护记录
  -> 答题区域 OCR
  -> 公共输出只保留脱敏答案
```

身份区域包括：

- `student_name`；
- `student_no`；
- `major`；
- `college`；
- `grade`。

身份记录示例：

```json
{
  "submission_id": "sub_001",
  "exam_id": "mayuan_1516",
  "identity_fields": {
    "student_name": {
      "value": "张三",
      "confidence": 0.91,
      "needs_review": false
    },
    "student_no": {
      "value": "202401001",
      "confidence": 0.95,
      "needs_review": false
    }
  },
  "access_scope": "internal_identity_only"
}
```

该记录不得被放入 `exam-document.v1` 公共结果，不得发送到批改服务或外部大模型。

如果姓名或学号识别失败：

- 不允许猜测；
- `value` 保持为空；
- `needs_review` 必须为 `true`；
- 添加 `identity_value_empty` 或 `low_identity_ocr_confidence`；
- 前端提示教师复核身份区域。

### 4.5 任务与复核管理层

新增内部任务记录，用于支持单份和批量处理：

```json
{
  "task_id": "task_001",
  "submission_id": "sub_001",
  "document_role": "answer_sheet",
  "status": "needs_review",
  "total_regions": 5,
  "completed_regions": 5,
  "needs_review_count": 2,
  "failed_count": 0,
  "error_code": null,
  "error_message": null
}
```

推荐状态：

```text
created
processing
needs_review
completed
failed
```

前端只需要查询任务记录，不需要读取内部目录结构。

复核记录建议保存：

```json
{
  "submission_id": "sub_001",
  "region_id": "q_1.1_p1_o1",
  "review_type": "answer_text",
  "before": "原始 OCR 文本",
  "after": "教师修正后的文本",
  "reviewed": true,
  "review_message": "已确认题号和答案区域"
}
```

身份区域复核和答案区域复核必须分开记录。

### 4.6 对外结果适配层

所有对外输出继续经过统一适配器生成，不允许各模块自行拼接结果。

试题卷继续输出：

```json
{
  "schema_version": "exam-document.v1",
  "submission_id": "sub_001",
  "exam_id": "mayuan_1516",
  "exam_name": "马克思主义原理15-16期末考试",
  "document_role": "question_paper",
  "layout_type": "free",
  "questions": []
}
```

答题卡继续输出：

```json
{
  "schema_version": "exam-document.v1",
  "submission_id": "sub_001",
  "exam_id": "mayuan_1516",
  "exam_name": "马克思主义原理15-16期末考试",
  "document_role": "answer_sheet",
  "layout_type": "fixed",
  "template_id": "tpl_mayuan_1516_v1",
  "template_name": "马克思主义原理15-16期末答题卡",
  "template_version": 1,
  "answers": []
}
```

新增的任务状态、内部路径、身份字段、数据库预留字段都不进入以上公共结构。

## 五、建议新增的内部模块

建议只增加内部模块，不重写现有识别模块：

```text
processing_registry/
  task_service.py          任务创建、状态更新、查询
  result_repository.py     结果保存和读取
  review_repository.py     复核记录保存和读取
  storage_adapter.py       文件存储适配

submission_registry/
  service.py                提交与考试、模板关系管理
  identity_repository.py   身份保护记录管理

api_service/
  status_routes.py         任务状态查询
  result_routes.py         结果查询
```

现有 `full_pipeline.py` 继续作为主流程入口，内部只增加：

1. 创建任务；
2. 更新任务状态；
3. 保存内部结果；
4. 保存复核状态；
5. 调用现有结果导出器生成公共输出。

## 六、接口兼容方案

### 6.1 不修改现有处理接口

当前处理请求继续使用：

- `file`；
- `document_role`；
- `exam_id`；
- `submission_id`；
- 现有模板和复核参数。

不新增必须字段，不删除字段，不修改原字段含义。

### 6.2 新增接口只作为补充能力

可以新增以下查询接口，但不影响原有调用方：

```text
GET /api/tasks/{task_id}
GET /api/submissions/{submission_id}
GET /api/submissions/{submission_id}/reviews
```

这些接口返回内部任务或复核状态。原有的处理接口仍然返回 `exam-document.v1`。

### 6.3 旧调用方兼容

旧调用方没有传 `submission_id` 时：

- 继续使用现有默认生成规则；
- 内部自动生成任务 ID；
- 不要求旧前端立即改动。

新前端建议主动传入稳定的 `submission_id`，以便查询结果和防止重复处理。

## 七、存储迁移方案

### 阶段一：继续使用 JSON 文件

保留当前文件型存储，新增独立文件：

```text
storage/registry/exams.json
storage/registry/exam_bindings.json
storage/registry/submissions.json
storage/registry/tasks.json
storage/registry/reviews.json
storage/registry/identity_records.json
storage/processing_results/
```

不要把所有内容继续塞进一个 JSON 文件，避免单文件越来越长且难以维护。

### 阶段二：抽象存储接口

代码只依赖以下内部方法：

```python
create_task(...)
update_task_status(...)
save_result(...)
get_result(...)
save_review(...)
get_identity_record(...)
```

当前实现可以是 JSON 文件，未来实现可以替换为 MySQL 或 PostgreSQL。

### 阶段三：接入正式业务系统

当整个产品后端数据库确定后，再由工作流或业务系统保存：

- 考试；
- 学生；
- 成绩；
- 教师；
- 批改状态。

识别模块只需要通过适配器写入或读取必要的 `exam_id`、`question_id` 和 `submission_id`，不需要重新设计公共输出。

## 八、错误和复核闭环

完整闭环应为：

```text
上传文件
  -> 文件校验
  -> 创建任务
  -> 解析和识别
  -> 生成候选区域
  -> 检查题号和模板
  -> 检查身份区域
  -> 生成脱敏结果
  -> 判断是否需要复核
  -> 教师复核
  -> 保存复核结果
  -> 重新导出 exam-document.v1
  -> 批改模块读取脱敏结果
```

错误必须面向教师使用自然语言表达，例如：

```text
未找到对应的考试，请先创建考试或检查考试名称。
答题卡模板还没有发布，暂时不能识别学生答题卡。
模板中的题号与试题卷不一致，请重新检查题号。
没有识别出姓名区域，请在模板复核页面重新划定姓名区域。
学号识别置信度较低，请放大图片后人工确认。
第 1.2 题出现重复区域，请删除多余区域后再发布模板。
```

错误响应可以继续使用现有错误响应结构，仅替换或补充中文 `message`，不改变成功输出结构。

## 九、批量处理方案

批量处理时，每一份答题卡仍然独立拥有一个 `submission_id`。

```text
批量任务 task_id
  -> submission_001
  -> submission_002
  -> submission_003
```

批量任务只做汇总，不改变单份结果格式：

```json
{
  "task_id": "batch_001",
  "status": "needs_review",
  "total": 30,
  "completed": 27,
  "needs_review": 3,
  "failed": 0
}
```

单份结果仍通过原有 `exam-document.v1` 返回。

## 十、与批改模块的边界

识别模块向批改模块只提供：

- `exam_id`；
- `exam_name`；
- `submission_id`；
- `question_id`；
- `question_no`；
- `question_text` 或 `answer_text`；
- 页码；
- 识别置信度；
- 是否需要复核；
- 风险标记。

识别模块不提供：

- 姓名；
- 学号；
- 专业；
- 学院；
- 年级；
- 原始身份区域图片；
- AI 分数；
- 最终分数。

姓名和学号只由后端内部绑定层使用，用于最终成绩匹配，不进入批改输入。

## 十一、实施顺序

### 第一步：冻结接口

建立接口回归样本，固定：

- 试题卷成功输出；
- 答题卡成功输出；
- 失败响应；
- 空白结果；
- 需要复核结果。

以 Schema 校验每次改动。

### 第二步：增加任务记录

在现有 `full_pipeline.py` 外围增加任务创建和状态更新，不修改识别算法。

验收：

- 每次处理都能查询任务状态；
- 处理失败能够保存中文错误原因；
- 重试不会生成重复任务结果。

### 第三步：增加结果仓储适配层

将当前结果保存和读取统一收口，不让 API、模板模块和识别模块分别直接操作文件。

验收：

- 结果可以通过 `submission_id` 查询；
- 原有输出完全一致；
- 替换存储实现时不需要修改处理流程。

### 第四步：完善复核记录

分别记录：

- 模板区域复核；
- 身份区域复核；
- 答案文本复核；
- 题号复核。

验收：

- 身份区域和答案区域不会混淆；
- 复核后重新导出结果；
- 复核信息不会进入批改输出中的敏感字段。

### 第五步：增加批量任务管理

只增加任务统计和查询，不改变单份文件处理接口。

验收：

- 能显示总数、成功数、失败数和待复核数；
- 单份失败不会阻断整个批量任务；
- 可重新处理失败提交。

### 第六步：与工作流进行适配

由工作流人员调用当前稳定接口，优先使用：

- `submission_id` 查询；
- `question_id` 对齐；
- `needs_review` 判断是否进入人工复核；
- `answer_text` 作为脱敏后的批改输入。

## 十二、测试和验收标准

### 12.1 接口兼容测试

- 原有请求可以继续成功处理；
- 原有成功响应通过 Schema 校验；
- 公共结果没有新增必填字段；
- `questions[]` 和 `answers[]` 的字段结构不变。

### 12.2 身份隔离测试

- 姓名区域不进入答案区域；
- 学号区域不进入答案区域；
- 姓名和学号不会出现在公共 JSON；
- 身份 OCR 失败时不猜测；
- 身份 OCR 低置信度时必须进入复核。

### 12.3 模板测试

- 草稿模板不能直接用于学生答题卡处理；
- 未发布模板不能绑定考试；
- 题目目录和模板 `question_id` 不一致时拒绝绑定；
- 已有提交后不能更换模板；
- 身份区域不能与答题区域重叠。

### 12.4 幂等测试

- 相同 `submission_id` 和相同文件重复提交，直接返回已保存结果；
- 相同 `submission_id` 但文件不同，返回中文错误；
- 相同文件但模板版本不同，不得错误复用旧结果。

### 12.5 真实样本测试

至少准备：

- 空白固定答题卡；
- 学生填写后的固定答题卡；
- 姓名和学号打印体样本；
- 姓名和学号手写体样本；
- 有旋转、阴影、裁边和模糊的边界样本；
- 自由版式主观题试题卷；
- 多页跨题试卷。

每份样本保存：

- 原始文件；
- 模板版本；
- 预期区域；
- 真实输出；
- 是否需要人工复核；
- 测试结论。

## 十三、风险控制

### 13.1 不改变接口风险

所有新增字段只进入内部任务和存储记录，公共输出由原有导出器生成。

### 13.2 存储迁移风险

先使用 JSON 文件验证业务流程，再通过仓储适配器迁移数据库，不直接重写主流程。

### 13.3 身份信息泄露风险

- 身份记录与答案记录分开保存；
- 批改适配器禁止读取身份字段；
- 日常结果查询不返回身份字段；
- 调试和错误信息不得打印姓名、学号和原始答案；
- 不使用外部 OCR 服务发送学生答卷，除非明确完成数据合规审批。

### 13.4 识别错误风险

- 低置信度必须标记复核；
- 不自动猜测姓名和学号；
- 不因 OCR 失败直接生成错误答案；
- 题号不一致时阻断后续批改输入。

## 十四、最终推荐方案

本项目下一阶段不需要复制参考阅卷系统的代码，而应在当前模块内部完成以下优化：

```text
保留现有识别算法
保留现有输入接口
保留 exam-document.v1
保留模板和 question_id 体系
保留身份隔离和脱敏流程

新增：
  任务状态管理
  结果查询仓储
  复核记录
  批量处理汇总
  内部适配层
  更完整的错误闭环
```

最终形成的模块边界为：

```text
识别模块：识别、切分、脱敏、复核、结果查询
批改模块：评分建议、人工批注、最终分数
业务系统：考试、学生、成绩、权限和导出
```

这种方案改动范围小、迁移风险低，能够继续兼容现有工作流，也为后续接入正式数据库和完整阅卷系统保留空间。
