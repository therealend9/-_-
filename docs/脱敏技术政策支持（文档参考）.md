# 识别模块数据清洗与脱敏技术说明

版本：v1.0  
整理日期：2026-07-19  
适用模块：考试文件识别模块  
代码位置：`d_module/data_redaction/`  
核心实现：`d_module/data_redaction/service.py`

## 1. 模块概述

本模块负责在试题卷、空白答题卡和学生答题卡完成识别、切分之后，对即将交给其他模块的结构化结果进行数据清洗和脱敏。

它的主要目标是：

> 让下游批改、分析或工作流模块尽量只收到完成清洗后的题目和答案，而不是直接收到姓名、学号、手机号、邮箱、身份证号、服务器路径等敏感信息。

本模块不负责：

- 判断学生真实身份；
- 将答题卡与学校学生名单绑定；
- 通过姓名上下文识别所有中文人名；
- 评分、生成批改建议或决定最终成绩；
- 替代服务器认证、数据库权限、文件加密和数据删除策略。

因此，当前实现应准确称为：

> 基于规则的识别结果数据清洗与脱敏模块。

不能表述为“已经完成了所有学生身份信息的自动识别和脱敏”。

## 2. 在整个识别流程中的位置

```text
上传试题卷/答题卡
    -> 文件安全检查
    -> 页面解析、OCR 和题目/答题区切分
    -> 生成 exam-document.v1 结构化结果
    -> 文件路径相对化
    -> 数据清洗与脱敏
    -> 写入脱敏审计记录
    -> 保存题目目录或处理结果
    -> API 返回脱敏结果
    -> 下游批改或工作流模块使用
```

数据清洗发生在 `full_pipeline.py` 中：

1. OCR 和题目切分先得到内部识别结果；
2. 服务器绝对路径先被转换或清理；
3. 调用 `redact_and_record()` 进行脱敏；
4. 脱敏后的结果才会保存到题目目录和处理结果；
5. API 返回的也是脱敏后的结果。

这意味着脱敏不是只在前端显示时进行，而是在后端最终数据交付边界进行。

## 3. 使用的主要技术

### 3.1 Python 递归遍历

代码通过递归方式遍历结果中的：

- 字典；
- 列表；
- 字符串；
- 数字、布尔值和空值。

它可以处理题目、答案、风险标记和分页信息等嵌套结构，而不需要为每一种 JSON 路径单独写处理代码。

处理原则是：

- 只修改字符串内容；
- 不改变 JSON 的键名；
- 不改变数组顺序；
- 不改变数字、布尔值和空值；
- 不改变 `exam-document.v1` 的整体结构。

### 3.2 正则表达式匹配

通过 Python `re` 模块识别格式比较明确的敏感内容：

- 手机号；
- 邮箱地址；
- 居民身份证号；
- 带标签的姓名，例如“姓名：张三”；
- 带标签的学号，例如“学号：202615001”；
- 带标签的身份证号；
- 文件名和服务器路径；
- 控制字符和不可见字符。

正则表达式适合处理格式固定的信息。例如手机号通常以 `1` 开头、身份证号具有固定长度、邮箱具有 `@` 和域名结构。

但正则表达式不适合单独判断所有中文姓名。例如：

```text
我认为张三的观点需要进一步讨论。
```

在没有姓名名单或明确标签的情况下，系统不会直接把“张三”判定为学生姓名，因为这样容易误伤正常答案内容。

### 3.3 身份证校验码验证

对于 18 位身份证号，系统不仅检查长度和字符格式，还会按照身份证校验位规则计算最后一位。

作用是减少把普通 18 位数字误认为身份证号的情况。

15 位身份证号由于格式较旧，当前实现按照固定长度进行脱敏。

### 3.4 字段名识别

除了检查文本内容，系统还会检查字段名。例如以下字段会被直接视为敏感字段：

```text
student_name
student_no
student_id
phone
mobile
email
id_card
origin_name
filename
```

如果结果中出现这些字段，即使字段里的内容没有通过正则匹配，也会被替换为统一占位符。

这种方式可以防止系统因为某种特殊格式没有被正则识别，而把明显属于身份信息的字段原样交给下游。

### 3.5 路径和文件名清洗

系统会清洗：

- 原始上传文件名；
- 原始文件名相关字段；
- `source_path`、`storage_path` 等路径字段；
- Windows 盘符路径；
- Linux 绝对路径；
- UNC 网络路径。

主要目的不是隐藏学生姓名，而是避免把服务器目录结构、用户本地目录、临时文件位置和部署信息泄露给前端或下游系统。

### 3.6 控制字符清除

系统会清除部分不可见控制字符、零宽字符、双向文本控制字符等。

作用包括：

- 防止日志显示异常；
- 防止文本方向被恶意改变；
- 减少隐藏字符绕过简单检查的机会；
- 避免 OCR 结果污染后续处理。

### 3.7 SHA-256 哈希

系统会对脱敏前后的结构化结果分别计算 SHA-256：

```text
source_result_sha256
redacted_result_sha256
```

哈希不是加密，也不能还原原文。它的作用是：

- 证明某次处理前后结果发生了变化；
- 识别结果是否被意外篡改；
- 支持问题追踪和处理记录核对；
- 在不保存原始答案文本的情况下，保留最小审计证据。

### 3.8 元数据审计

脱敏报告保存以下信息：

- `submission_id`；
- 脱敏规则版本；
- 脱敏状态；
- 命中的敏感信息类别；
- 替换次数；
- 脱敏前后的结果哈希；
- 处理时间。

报告不保存：

- 原始姓名；
- 原始学号；
- 原始手机号；
- 原始邮箱；
- 原始答案内容；
- 原始文件名和服务器路径。

审计文件名使用 `submission_id` 的 SHA-256 生成，避免直接把提交标识作为文件名暴露。

### 3.9 原子文件写入

审计报告先写入临时文件，写入完成后再替换正式文件：

```text
report.json.tmp -> report.json
```

这样可以避免程序中途退出时留下一个不完整的 JSON 文件。

### 3.10 接口结构兼容

清洗模块采用“只改值、不改结构”的方式：

- 不新增接口参数；
- 不删除接口字段；
- 不改变 `questions[]` 和 `answers[]` 的层级；
- 不改变 `question_id`、`question_no` 的字段名称；
- 不改变 `exam-document.v1` 的版本。

例如：

```json
{
  "question_id": "1.1",
  "answer_text": "姓名：张三；学号：202615001"
}
```

清洗后变为：

```json
{
  "question_id": "1.1",
  "answer_text": "姓名：[REDACTED]；学号：[REDACTED]"
}
```

字段仍然存在，只有敏感字符串被替换。

## 4. 当前支持的清洗规则

| 原始内容 | 当前处理方式 | 输出示例 |
|---|---|---|
| 手机号 | 正则匹配 | `[PHONE_REDACTED]` |
| 邮箱 | 正则匹配 | `[EMAIL_REDACTED]` |
| 有效 18 位身份证号 | 格式和校验码验证后替换 | `[ID_CARD_REDACTED]` |
| 15 位身份证号 | 固定格式替换 | `[ID_CARD_REDACTED]` |
| `姓名：张三` | 标签和值一起识别 | `姓名：[REDACTED]` |
| `学生姓名：张三` | 标签和值一起识别 | `学生姓名：[REDACTED]` |
| `学号：202615001` | 标签和值一起识别 | `学号：[REDACTED]` |
| `student_id: A2026001` | 标签和值一起识别 | `student_id:[REDACTED]` |
| 原始文件名字段 | 按字段名替换 | `[REDACTED]` |
| 服务器绝对路径 | 按字段名或路径格式替换 | `[PATH_REDACTED]` |
| 控制字符 | 删除 | 清洗后的文本 |

官方出处：中国政府网  
https://www.gov.cn/xinwen/2021-08/21/content_5632486.htm

与本模块相关的政策要求：

- 处理个人信息应当遵循合法、正当、必要和诚信原则；
- 处理目的应当明确，处理范围应当与目的直接相关；
- 个人信息处理者应采取措施防止未经授权访问、泄露、篡改和丢失；
- 对敏感个人信息、委托处理、跨境提供等情况，应按适用条件进行更严格管理和影响评估；
- 发生个人信息泄露等安全事件时，应采取补救措施并履行相应通知义务。

对本模块的技术意义：

1. 将脱敏放在结果保存和对外返回之前，体现“最小化提供数据”；
2. 不把姓名、学号等信息用于批改提示词、模型训练或无关分析；
3. 审计只保存哈希和统计信息，减少审计记录再次泄露原文的风险；
4. 通过固定占位符，让下游得到完成业务所需的题目和答案，而不是完整身份信息。

## 6. 这些技术分别解决什么问题

| 技术 | 解决的问题 | 具体意义 |
|---|---|---|
| 正则表达式 | 识别固定格式的手机号、邮箱、身份证号等 | 快速处理高确定性敏感信息 |
| 字段名匹配 | 防止明显的身份字段被原样传递 | 即使值格式特殊，也能按字段保护 |
| 身份证校验码 | 减少普通数字被误判为身份证号 | 降低误杀题目内容的概率 |
| 路径清洗 | 防止暴露服务器和本地目录 | 保护部署信息和内部文件结构 |
| 控制字符清洗 | 防止隐藏字符污染结果或日志 | 提高文本安全性和可处理性 |
| 递归遍历 | 处理嵌套的题目和答案结构 | 保证不同接口结果都能统一清洗 |
| 固定占位符 | 替换原始敏感内容 | 下游仍能识别“这里有信息，但不需要知道原文” |
| SHA-256 | 记录前后结果是否变化 | 在不保存原文的情况下提供核验依据 |
| 原子写入 | 避免审计文件写一半 | 防止生成损坏的审计记录 |
| 失败阻断 | 脱敏或审计失败时不继续交付 | 避免未清洗数据意外进入下游 |

## 7. 当前实现的安全价值

### 7.1 降低数据扩散范围

姓名、学号等信息只要被识别模块清洗，后续批改模块就不需要接收这些内容。参与系统的模块越多，数据传播范围越小，泄露风险越低。

### 7.2 保持现有接口不变

下游不需要重新学习一套接口。原有字段和层级继续保留，敏感内容只被替换成占位符。

### 7.3 减少审计记录泄露

审计报告不保存原始答案和身份信息，只保存统计信息和哈希，因此即使审计文件被查看，也不会直接暴露学生原文。

### 7.4 降低外部模型使用风险

如果下游需要调用外部大模型，经过清洗的结果可以作为更安全的输入边界，减少学生身份和服务器信息被发送到外部平台的可能。

### 7.5 支持问题追踪

规则版本、命中类别、数量、时间和哈希可以帮助定位：

- 哪一版规则处理了结果；
- 哪一类敏感信息被替换；
- 清洗前后结果是否发生变化；
- 某次结果是否被重复处理或篡改。

## 8. 当前不能保证的内容

当前模块不能保证识别所有敏感信息，特别是：

- 没有标签的普通中文姓名；
- OCR 识别错误后的姓名和学号；
- 学校自定义的特殊学号格式；
- 家庭住址和家庭成员信息；
- 思想汇报、心理健康和处分等上下文敏感内容；
- 图片中仍然存在但没有进入结构化文本的姓名或学号；
- 原始文件本身的存储加密和权限安全。

因此，若要做到“答题卡上的姓名、学号区域一定不进入下游”，还需要结合：

1. 固定答题卡中的姓名区和学号区配置；
2. 姓名区、学号区与答题区的空间隔离；
3. 模板发布前人工复核；
4. 原始图像区域裁剪或遮盖；
5. 低置信度结果进入人工复核；
6. 原始图片和身份区域的独立权限控制。

这些属于身份区域隔离和平台安全闭环，不是当前规则型文本脱敏单独能够解决的问题。

## 9. 当前测试依据

当前数据清洗模块已有测试覆盖：

- 带标签的姓名和学号清洗；
- 手机号清洗；
- 邮箱清洗；
- 身份证号清洗；
- OCR 将邮箱和中文标签拼接时的清洗；
- 文件名和绝对路径清洗；
- 控制字符处理；
- 不修改原始输入对象；
- 审计报告不保存原始姓名；
- 普通答案中的无标签姓名不被误判。

```text

```

后续还需要使用经过授权的真实样本或高质量模拟样本，测量漏检率和误杀率，尤其要测试手写 OCR 错误、姓名区漏裁剪、学号位数变化和中英文混排情况。

## 10. 可用于项目文档的技术表述

### 简短表述

本项目在考试文件识别结果交付前增加了基于规则的数据清洗与脱敏模块，使用递归 JSON 遍历、正则表达式、身份证校验码验证、路径清洗、控制字符清除、SHA-256 哈希和元数据审计等技术，对手机号、邮箱、身份证号、带标签的姓名和学号、文件名及服务器路径进行处理。该模块不改变既有接口结构，在不影响下游系统对题目和答案字段使用的前提下，降低个人信息和内部路径向外部模块扩散的风险。

### 完整表述

依据《中华人民共和国个人信息保护法》关于合法、正当、必要和最小化处理的要求，以及《中华人民共和国数据安全法》《中华人民共和国网络安全法》关于数据安全保护和网络安全防护的要求，本项目将数据清洗设置在识别结果持久化和对外返回之前。系统对结构化识别结果进行递归遍历，对具有明确格式或明确字段标识的个人信息进行替换，对服务器路径和控制字符进行清理，并仅保存规则版本、命中类型、数量、处理时间和哈希等最小化审计信息。对于脱敏失败或审计写入失败的情况，系统阻止结果继续交付，从而为下游批改工作流提供经过清洗的数据边界。

## 11. 与其他安全工作的关系

本模块是安全链路中的“结果交付前清洗环节”，不能替代以下措施：

- API 登录、角色权限和对象级访问控制；
- 原始文件加密和私有存储；
- 模板原图访问控制；
- 数据留存和自动删除；
- 病毒扫描和文件沙箱；
- 安全事件响应；
- 外部模型供应商审查；
- 定期个人信息保护合规审计。

完整安全链路应为：

```text
身份认证与授权
  -> 安全上传
  -> 文件真实性和资源检查
  -> OCR 与切分
  -> 身份区域/答题区域隔离
  -> 数据清洗与脱敏
  -> Schema 和风险校验
  -> 只交付脱敏结果
  -> 访问审计
  -> 按策略删除和复核
```
