# 当前版本数据脱敏迁移实施方案

## 1. 目标与边界

本次只建设考试文件识别模块的结果数据脱敏能力，不建设姓名、学号 OCR，不接入考试名单，不建立学生与答题卡绑定，也不涉及评分、批改建议或成绩。

对外接口保持不变：

- `POST /api/process` 的 `file`、`document_role`、`exam_id`、`submission_id` 不变。
- `GET /api/submissions/{submission_id}` 不变。
- 成功结果继续为 `exam-document.v1`，不新增或删除任何字段。
- `questions[]` 和 `answers[]` 的字段、层级、`question_id` 与 `question_no` 均不变；只有命中敏感规则的字符串值会被替换为固定占位符。

本次产物能处理高确定性、规则可验证的信息：手机号、邮箱、有效居民身份证号、明确标注的姓名和学号、输出中的原始文件名、服务器绝对路径、控制字符和双向控制字符。

它不能在无名单、无身份区域模型、无真实样本的条件下可靠识别任意中文人名、所有学号格式、家庭住址、思想汇报或其他上下文隐私信息。因此本次只能称为“规则型输出脱敏”，不能称为“已完成学生身份脱敏闭环”。

## 2. 可行性评估

当前实现具备接入条件：识别结果已在 `full_pipeline.py` 汇总为单个对象，处理结果已有统一持久化入口，且 API 只返回最终结果。因此可以在一个边界统一处理所有输出，而不改 OCR、试题切分、答题区裁剪、模板或接口参数。

风险和限制如下：

| 风险 | 当前控制 | 剩余限制 |
|---|---|---|
| 手机号、邮箱、身份证号出现在答案正文 | 正则与身份证校验位验证后替换 | OCR 错字可能使格式失效 |
| 明确写出的“姓名：”“学号：” | 仅替换标签后的值 | 不识别无标签姓名或学校自定义学号格式 |
| 调试输出中的文件名和绝对路径 | 按字段名和绝对路径模式替换 | 不处理原始文件存储权限，仍由上传安全和部署权限负责 |
| 脱敏规则故障 | 脱敏或审计写入失败即中止处理，不保存和返回结果 | 需要后续用真实授权样本评估漏检率和误杀率 |
| 与现有调用方兼容 | 不新增响应字段；现有字段类型不变 | 命中规则的文本会变为占位符，下游必须把它当作普通文本处理 |

不需要新增 OCR 模型、数据库或外部服务。实现技术为 Python 标准库：深拷贝式递归遍历、`re`、SHA-256、原子 JSON 写入和文件系统权限隔离。这样可在当前运行环境直接部署，且不会把任何内容送往外部模型。

## 3. 目标链路

```text
安全上传
  -> OCR 与试题/答题区切分
  -> 构造 exam-document.v1 结果
  -> 路径相对化
  -> 规则型脱敏
  -> 写入不含原文的脱敏审计记录
  -> 保存题目目录或答题卡提交记录
  -> 保存最终处理结果
  -> API 返回 / 查询接口读取同一份脱敏结果
```

脱敏位于所有结果持久化之前，保证以下位置不会保存未脱敏的可交付 OCR 文本：

- `storage/processing_results/` 的最终结果文件；
- 试题卷对应考试的题目目录；
- `POST /api/process` 的成功响应；
- `GET /api/submissions/{submission_id}` 的查询响应；
- 仅在内部调用时请求的中间结果中的文件路径和 OCR 文本。

原始上传文件仍保留在受控存储区，属于上传安全与存储权限的管理范围；本次不把它们复制到脱敏目录，也不对外暴露。

## 4. 分阶段迁移路径

### 阶段 A：代码接入与本地验证

1. 新增 `d_module/data_redaction/`，实现纯函数 `redact_result()`。
2. 规则处理只修改字符串值，不改变 JSON 键、数组顺序、数值、布尔值和空值。
3. 新增 `redact_and_record()`，在 `storage/redaction/` 写入仅含规则版本、命中类别、数量、结果哈希和时间的审计记录；审计记录禁止保存原始 OCR 文本与替换前值。
4. 在 `full_pipeline.py` 的路径相对化之后、任何结果持久化之前调用脱敏函数。
5. 为手机、邮箱、身份证、显式姓名/学号、路径、普通中文答案、幂等重试和持久化查询编写回归测试。

验收：脱敏前后均满足既有 `exam-document.v1` 结构；接口输入输出字段不变；命中内容不会进入最终结果和题目目录。

### 阶段 B：预生产灰度

1. 使用学校明确授权的脱敏或模拟样本运行，不使用真实学生身份数据。
2. 人工检查命中报告，不检查或导出报告以外的原文。
3. 统计规则命中率、误替换和漏检例子，调整规则版本。
4. 仅在校内受控环境中部署，确认 `storage/redaction/` 与结果目录均不对外静态暴露。

验收：规则没有错误替换常见题干、题号、年份、分数或普通理论表述；脱敏失败时 API 不返回成功结果。

### 阶段 C：生产启用

1. 先备份当前结果目录和索引，不覆盖既有历史结果。
2. 发布包含脱敏模块的版本；新提交自动走脱敏链路。
3. 旧提交维持原有查询行为，不在无审批的情况下批量重写。
4. 若需重新处理旧提交，使用新的 `submission_id` 或完成独立迁移审计，避免破坏既有幂等指纹。

回滚：撤销 `full_pipeline.py` 的单一脱敏调用并保留历史结果目录，不删除原始文件、不批量修改已保存结果。由于脱敏报告不含原文，删除报告不会影响识别重跑，但会损失审计证据，生产环境不应这样做。

### 阶段 D：后续身份闭环（不属于本次实现）

在有授权模板、名单、真实样本和人工复核机制后，再单独实施：身份区域配置、姓名/学号本地 OCR、名单精确匹配、冲突复核、身份映射加密存储和基于名单的姓名脱敏。该阶段需要新的内部数据表和权限域，但仍不应改变 `exam-document.v1`。

## 5. 占位符与审计规则

| 命中内容 | 输出替换 |
|---|---|
| 手机号 | `[PHONE_REDACTED]` |
| 邮箱 | `[EMAIL_REDACTED]` |
| 有效身份证号 | `[ID_CARD_REDACTED]` |
| `姓名：张三` | `姓名：[REDACTED]` |
| `学号：2026...` | `学号：[REDACTED]` |
| 原始文件名 | `[REDACTED]` |
| 服务器绝对路径 | `[PATH_REDACTED]` |

内部审计记录使用 `submission_id` 的 SHA-256 作为文件名，内容只包含：`submission_id`、规则版本、状态、命中类别、命中数量、脱敏前后 JSON 哈希和时间。记录不保存手机号、姓名、学号、答案原文、文件名或路径。

## 6. 当前交付标准

1. 所有新处理结果在持久化和返回前完成规则型脱敏。
2. 题目目录与最终结果不保留命中的敏感文本。
3. 脱敏过程或审计写入异常时请求失败，不返回未处理的内容。
4. 相同 `submission_id` 与相同输入仍命中现有幂等结果。
5. 已有接口字段和 JSON Schema 不发生结构变化。
6. 文档和代码均明确声明：本次不提供姓名学号识别、学生绑定或任意中文姓名识别能力。
