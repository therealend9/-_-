# 考试阅卷产品三人协作任务分工与开发测试计划

## 1. 目标与范围

本阶段围绕大学思政课期末考试，完成从试卷和答题卡进入系统，到按题目组织材料并交给批改模块的完整链路。

当前识别模块只负责三类输入：

- 试题卷：识别自由版式试题，输出 `questions[]`。
- 空白答题卡：生成、复核、发布固定答题卡模板。
- 学生答题卡：按已发布模板定位答题区域，识别学生作答，输出 `answers[]`。

批改模块由原负责人继续负责，本计划不调整其负责人，也不把评分规则迁移到识别模块。

本阶段不纳入：独立标准答案文件识别与存储、客观题识别、自动反作弊判断、数据库重构和部署治理。

## 2. 图片需求转化

从当前需求截图中可以确认以下产品方向：

- 支持批量处理试卷。
- 使用答题卡模板和学生作答结果组织批改材料。
- 按试卷中的题目进行分题。
- 用户可以点击某一道题查看该题相关内容。
- 需要有队列/列表来承载待处理内容和批改建议。

截图中部分文字被遮挡，以下内容需要产品负责人后续确认，不作为本阶段强制实现项：

- “两个队列”的准确名称和流转规则。
- “建议”是批改模块生成的评分建议、人工批注，还是其他业务信息。
- 截图中的学生回答标识和异常检测描述是否属于当前版本。

## 3. 三人固定分工

### 成员一：批改模块负责人（保持不变）

#### 开发职责

1. 维护现有批改模块的核心规则、批改流程和评分结果格式。
2. 明确批改模块接收的最小输入：
   - `exam_id`、`exam_name`；
   - `question_id`、`question_no`、`question_text`；
   - `answer_text`、`is_blank`；
   - 原始图片/裁剪区域引用；
   - 识别置信度和 `needs_review`。
3. 提供按 `question_id` 接收识别结果的适配层，不允许通过数组下标或印刷题号关联。
4. 实现批改队列状态，例如待批改、批改中、待人工确认、已完成、异常。
5. 实现题目详情所需的批改结果和建议数据结构。
6. 约定人工修改分数、批注、建议后的保存和重新打开行为。

#### 测试目标

- 相同 `question_id` 的试题和学生作答能够进入同一道题的批改流程。
- 空白答题区不会被误判为有效作答。
- 手写识别结果为空或低置信度时能够进入人工复核队列。
- 批改结果重复提交不会产生重复记录。
- 页面刷新、重新进入和批量处理后，批改状态保持一致。
- 使用固定的模拟输入完成批改模块单元测试和接口契约测试。

#### 交付物

- 批改模块输入/输出接口文档。
- `question_id` 对齐的适配代码。
- 批改队列和题目详情接口。
- 批改模块单元测试、契约测试和一份可供联调用的模拟数据。

#### 不负责

- 不修改识别模块的 OCR、模板检测和页面对齐逻辑。
- 不负责前端页面布局和上传组件。

### 成员二：试卷与答题卡识别后端负责人

#### 开发职责

1. 维护 `question_paper` 自由版式试题识别流程。
2. 维护空白答题卡模板流程：
   - 上传空白答题卡；
   - 自动提出答题区域；
   - 保存人工复核结果；
   - 发布模板并绑定考试。
3. 维护 `answer_sheet` 学生答题卡流程：
   - 页面渲染和对齐；
   - 固定区域裁剪；
   - 空白检测；
   - 手写 OCR；
   - 同一题多个区域合并。
4. 保证试题卷、答题卡模板和学生结果使用同一套 `question_id`。
5. 维护 `exam-document.v1` Schema 和 API 文档。
6. 提供前端所需的错误码、处理状态和人工复核字段。
7. 根据批改负责人的输入契约提供识别结果适配，不在识别模块中实现评分。

#### 测试目标

- 空白答题卡能够生成模板草稿，并允许完整保存老师调整后的区域。
- 模板未发布、考试不存在、题目 ID 不一致、模板版本缺失时返回明确错误。
- 已有学生提交后更换模板会被拒绝。
- 试题卷能够返回题目文本、题号、页码、置信度和复核标记。
- 学生答题卡能够按模板返回 `answers[]`，包括空白区和手写区。
- 覆盖多页答题卡、同题多区域、页面偏移、旋转、空白答题区和低置信度场景。
- 使用真实或脱敏样本保存输入、模板、裁剪结果和最终 JSON，形成回归集。
- API 层拒绝不支持的 `answer_key`、客观题专用流程和未绑定模板请求。

#### 交付物

- 可运行的识别 API 和命令行入口。
- `exam-document.v1` Schema、接口文档和错误码说明。
- 空白答题卡、学生答题卡、试题卷测试样本及结果。
- 单元测试、API 测试、回归测试报告。

#### 不负责

- 不实现分数计算、标准答案比对和批改建议生成。
- 不直接决定批改页面的视觉交互。

### 成员三：前端页面与联调测试负责人

#### 开发职责

1. 实现考试列表页：
   - 显示 `exam_id` 对应的考试名称；
   - 显示考试状态、模板名称、模板版本；
   - 进入试题、模板、学生答题卡和批改工作区。
2. 实现考试创建与试题卷导入页：
   - 创建考试；
   - 上传试题卷；
   - 展示识别出的题目列表；
   - 标记 `needs_review` 题目；
   - 展示题号、题干、页码和置信度。
3. 实现空白答题卡模板页：
   - 上传空白答题卡；
   - 展示逐页图片；
   - 显示自动检测的答题区域；
   - 支持移动、缩放、删除和新增区域；
   - 为区域选择题目目录中的 `question_id`；
   - 显示 `needs_review` 状态；
   - 发布模板和绑定考试前展示二次确认。
4. 实现学生答题卡批量导入页：
   - 批量选择文件；
   - 选择考试，不允许直接选择任意模板；
   - 展示单个文件的处理中、成功、需复核、失败状态；
   - 展示每道题的裁剪图、识别文本、空白状态、置信度和风险标记。
5. 实现按题目进入批改工作区的入口：
   - 按 `question_id` 聚合试题和学生作答；
   - 支持点击题目查看原题、学生答案和批改建议；
   - 对接批改模块负责人提供的队列和结果接口。
6. 实现空状态、加载中、上传失败、OCR 依赖异常、模板未绑定、低置信度和人工复核状态。

#### 测试目标

- 教师可以从创建考试开始，完成试题卷导入、模板发布、学生答题卡导入和进入批改页。
- 前端显示考试名称，而不是只显示难以理解的 `exam_id`。
- 模板编辑器保存后重新打开，区域位置、题号和版本不丢失。
- 前端不会把 `question_no`、数组下标或模板印刷题号当作关联键。
- 批量上传中单个文件失败不会阻塞其他文件，并能重试失败项。
- 题目详情中的试题、学生答案和批改建议不会串题。
- 对桌面端常用分辨率完成视觉和交互测试，并至少覆盖窄屏布局的关键操作。
- 使用后端模拟数据完成前端自动化测试，再使用真实样本完成一次端到端测试。

#### 交付物

- 考试管理、试题导入、模板复核、学生答题卡导入、按题批改入口页面。
- API 客户端和统一错误处理。
- 前端组件测试、页面流程测试和端到端测试报告。
- 联调问题清单和最终操作说明。

#### 不负责

- 不修改批改规则和分数计算。
- 不绕过后端模板绑定、版本和题目 ID 校验。

## 4. 接口与代码 ownership

| 范围 | 主要负责人 | 其他人协作方式 |
|---|---|---|
| `b_module/` 批改相关代码 | 成员一 | 成员二提供识别结果适配，成员三只通过接口调用 |
| `d_module/`、`b_module/question_segmenter/`、`template_builder/`、`template_registry/` | 成员二 | 成员三提供前端字段需求，成员一提供输入契约 |
| `api_service/` 识别接口 | 成员二 | 成员三联调；批改接口由成员一负责 |
| 前端应用目录 | 成员三 | 成员一确认批改交互，成员二确认识别交互 |
| `contracts/` 和接口文档 | 成员二主维护 | 成员一确认批改输入，成员三确认前端可用性 |
| `test_artifacts/` 真实样本和端到端结果 | 成员二维护 | 成员三执行页面测试，成员一执行批改回归 |

同一文件或模块必须由一个人负责合并，其他人通过 issue、接口文档或独立分支提交修改建议，避免互相覆盖。

## 5. 共同接口约定

识别模块到批改模块的最小数据关系：

```text
questions[].question_id
answers[].question_id
批改记录.question_id
```

学生答题卡结果至少包含：

```json
{
  "exam_id": "mayuan_1516",
  "submission_id": "student_001",
  "question_id": "2.1",
  "question_no": "2.1",
  "answer_text": "学生作答文本",
  "is_blank": false,
  "confidence": 0.83,
  "needs_review": false,
  "risk_flags": []
}
```

批改模块不得依赖识别模块内部中间产物；前端不得自行拼接或重新生成 `question_id`。

## 6. 分阶段计划

### 阶段一：接口冻结与任务准备

成员一提交批改输入/输出字段；成员二确认识别 API 和 Schema；成员三根据字段建立页面路由和模拟数据结构。

完成标准：三方确认字段含义、错误状态和 `question_id` 规则，后续变更必须记录在接口文档中。

### 阶段二：后端和页面并行开发

成员一完成批改队列和题目详情接口；成员二完成试题、模板、学生答题卡链路和测试样本；成员三完成考试管理、模板编辑和批量导入页面。

完成标准：前端可以使用模拟数据跑通全部页面，后端可以使用命令行跑通三类输入。

### 阶段三：识别与前端联调

成员三接入成员二的真实 API；成员一接入题目和学生答案数据；成员二根据联调结果修复字段、错误码和处理状态问题。

完成标准：从试题卷创建考试，到模板绑定，再到学生答题卡进入批改工作区，至少完成一条真实样本链路。

### 阶段四：联合验收

三人共同执行：正常流程、空白答题卡、已填写答题卡、低置信度、模板未绑定、题目 ID 不一致、批量文件部分失败、重复提交和页面刷新恢复测试。

完成标准：阻断性问题为零；高风险结果能够进入人工复核；所有失败请求有明确提示；输出 JSON 和批改入参通过契约校验。

- [ ] 成员三完成页面骨架和模拟数据联调。

- [ ] 三方确认“两个队列”的准确名称、状态转换和权限。
- [ ] 三方确认批改建议的来源、字段和是否允许人工修改。
- [ ] 明确真实测试样本的存储位置和脱敏要求。

## 8. 最终验收指标

1. 三类输入均有可重复的操作路径和测试记录。
2. 试题、模板区域、学生答案和批改记录的 `question_id` 100% 对齐。
3. 模板编辑、发布、绑定和更换限制均有前端提示与后端校验。
4. 低置信度、空白区域、对齐失败和 OCR 失败均可人工复核。
5. 批改负责人可以不改动核心批改逻辑，直接消费识别模块输出。
6. 前端不依赖后端内部中间产物，只依赖正式接口和 Schema。
