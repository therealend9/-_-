# 考试文件上传安全实施方案

版本：v1.0  
适用模块：考试文件识别模块  
适用输入：试题卷、空白答题卡、学生答题卡  
不包含：评分逻辑、标准答案管理、批改模型内部安全

## 实施状态

截至当前版本，P0 中以下能力已实现：

- `POST /api/process` 和 `POST /api/templates/drafts` 统一进入分块隔离上传服务
- 文件上传不再由 API 一次性读取为内存字节串
- 扩展名、客户端 MIME、文件头和实际格式三重校验
- PNG/JPG 尺寸和像素限制
- PDF 打开、页数、页面尺寸和按 OCR DPI 计算的单页渲染像素限制
- DOCX ZIP 条目数、解压总大小、压缩比、路径穿越、宏和外部关系限制
- 隔离区临时文件在成功放行或校验失败后清理
- 已放行文件保存 SHA-256 和安全清单
- 成功处理接口保持 `exam-document.v1` 不变，安全失败返回结构化错误码

以下能力仍为后续 P1/P2：ClamAV 扫描、容器化解析沙箱、上传频率限制、审计事件持久化、隔离区定时清理和加密对象存储。

## 1. 建设目标

本方案用于保护文件从上传到 OCR、模板识别和结果持久化的完整链路，解决以下问题：

- 恶意文件伪装成 PDF、DOCX 或图片
- 超大文件、超大图片和 PDF 炸弹消耗内存、CPU 或磁盘
- DOCX 压缩包解压时发生路径穿越或解压炸弹
- 文件内容在未完成安全检查前被 OCR 或文档转换组件处理
- 上传文件被覆盖、篡改或无法追溯
- 模板上传接口和普通处理接口使用不同安全规则
- 安全失败后产生临时文件、数据库记录或半成品结果

安全设计目标是：

```text
文件接收
 -> 限制请求资源
 -> 隔离区落盘
 -> 真实类型校验
 -> 格式专项检查
 -> 病毒/恶意内容扫描
 -> 文件标准化
 -> 放行到正式存储
 -> OCR/模板处理
 -> 结果持久化
 -> 审计和过期清理
```

只有通过安全检查的文件才能进入文档解析、页面渲染和 OCR 流程。

## 2. 当前系统边界

当前主要入口：

| 入口 | 当前用途 | 安全改造要求 |
|---|---|---|
| `POST /api/process` | 处理试题卷和学生答题卡 | 必须经过统一安全上传服务 |
| `POST /api/templates/drafts` | 上传空白答题卡并生成模板草稿 | 必须使用与普通文件相同的校验流程 |
| `d_module.file_ingest.service.create_file_task` | 创建文件任务并保存原始文件 | 作为正式文件放行后的唯一文件任务入口 |
| `d_module.file_ingest.validators` | 扩展名、MIME、大小校验 | 扩展为文件头、压缩包、图片、PDF专项校验 |
| `d_module.file_ingest.storage` | 保存原始文件 | 增加隔离区、正式区、不可变文件名和清理策略 |
| `processing_registry` | 保存最终处理结果 | 只允许安全放行后的文件进入结果流程 |

当前代码中的 `UploadFile.read()` 会一次性读取全部上传内容；模板上传还会先保存文件，再由后续模块处理。安全改造后，两个入口必须统一调用 `secure_upload`，禁止自行读取和保存上传内容。

## 3. 威胁模型

### 3.1 外部攻击者

- 上传伪造扩展名的可执行文件、脚本或恶意压缩包
- 上传损坏 PDF、DOCX 或图片触发解析器漏洞
- 上传超大文件、超大页面或高压缩比文件造成拒绝服务
- 使用 `../`、绝对路径或特殊文件名覆盖服务器文件
- 重复上传大量文件消耗磁盘
- 构造恶意内容影响后续 AI 工作流

### 3.2 普通业务用户误操作

- 上传错误类型文件
- 上传未完成或损坏的扫描文件
- 上传页数过多或分辨率过高的原始扫描件
- 重复提交同一文件

### 3.3 被入侵的上游系统

- 伪造 `Content-Type`
- 传递错误的原始文件名
- 重放旧的 `submission_id`
- 通过接口参数指定未授权模板或路径

### 3.4 下游解析器风险

- PDF 渲染器、LibreOffice、DOCX 解析器和图像库的漏洞
- 文档外部链接、宏、嵌入对象和 JavaScript
- OCR 文本中的提示词注入、控制字符或超长内容影响批改工作流

## 4. 总体架构

### 4.1 目录分层

```text
storage/
  quarantine/
    {upload_id}/
      payload.part
      upload.json
      security.json
  accepted/
    {file_id}/
      original.{ext}
      normalized.{ext}
      manifest.json
  raw/
    {file_id}.{ext}
  pages/
  converted/
  docx_images/
  processing_results/
  audit/
```

规则：

1. `quarantine` 中的文件不能被 OCR、PDF 渲染或 DOCX 解析模块读取。
2. 只有状态变为 `accepted` 后，才能创建 `FileTask`。
3. `raw`、`pages`、`converted` 和 `docx_images` 只保存安全放行后的文件。
4. 所有路径由服务端根据随机 ID 生成，禁止使用用户原始文件名拼接路径。
5. `quarantine` 文件必须在成功放行、拒绝或超时后清理。

### 4.2 状态机

```text
RECEIVING
  -> QUARANTINED
  -> SIGNATURE_CHECKED
  -> FORMAT_CHECKED
  -> RESOURCE_CHECKED
  -> MALWARE_SCANNED
  -> ACCEPTED
  -> PROCESSING
  -> COMPLETED
```

失败状态：

```text
RECEIVING      -> REJECTED
QUARANTINED    -> REJECTED
任意检查状态    -> SCAN_ERROR
PROCESSING     -> PROCESSING_FAILED
```

状态转移必须满足：

- 状态只能向前推进，不能从 `REJECTED` 回到 `ACCEPTED`
- 每次状态转移记录时间、原因和操作者
- `ACCEPTED` 之前不能创建 `FileTask`
- `COMPLETED` 之前不能写入最终处理结果
- 失败状态触发临时文件清理，但保留最小化审计记录

## 5. 统一上传流程

### 5.1 请求入口

`POST /api/process` 和 `POST /api/templates/drafts` 统一调用：

```text
secure_upload(
    upload_file,
    upload_purpose,
    submission_id,
    exam_id,
)
```

`upload_purpose` 取值：

- `question_paper`
- `answer_sheet`
- `answer_sheet_template`

接口成功响应的 `exam-document.v1` 结构保持不变。安全上传的内部状态、扫描信息和文件哈希写入内部清单和审计记录，不直接改变现有识别结果字段。

### 5.2 分块接收

禁止直接使用：

```python
content = await file.read()
```

改为固定块大小读取：

```text
chunk_size = 1 MB
max_file_size = 50 MB
```

每读取一块执行：

1. 累加已接收字节数
2. 更新 SHA-256
3. 写入隔离区临时文件
4. 超过大小上限立即停止并删除临时文件

同时设置 HTTP 层请求体限制，避免应用层尚未接管前就被大请求耗尽资源。

### 5.3 原始文件名处理

原始文件名只保存为展示字段，不用于路径：

- 去除控制字符和换行符
- 限制长度为 256 个字符
- 只允许文件名，不允许 `/`、`\\`、盘符和 UNC 路径
- 规范化 Unicode 文件名
- 存储路径使用 `file_id` 和服务端确定的扩展名

## 6. 文件真实性校验

### 6.1 三重校验

文件必须同时通过：

```text
扩展名允许
 + 客户端 MIME 可接受
 + 文件头和实际解析类型一致
```

客户端 MIME 只能作为提示，不能作为最终判断。

### 6.2 类型规则

| 类型 | 文件头/解析检查 | 后续检查 |
|---|---|---|
| PNG | PNG 签名 + Pillow/OpenCV 解码 | 尺寸、像素、通道、解码内存 |
| JPG/JPEG | JPEG 签名 + Pillow/OpenCV 解码 | 尺寸、像素、通道、解码内存 |
| PDF | `%PDF-` 签名 + PyMuPDF 打开 | 页数、页面尺寸、嵌入对象、渲染限制 |
| DOCX | ZIP 签名 + `[Content_Types].xml` + `word/document.xml` | 解压大小、文件数、路径、宏、外链 |

拒绝以下情况：

- 扩展名与真实类型不一致
- 文件头正确但解析器无法打开
- 图片实际为超大像素图
- PDF 或 DOCX 结构不完整
- MIME 为可执行文件、脚本或未知类型

错误码：

```text
FILE_EXTENSION_NOT_ALLOWED
FILE_MIME_NOT_ALLOWED
FILE_SIGNATURE_MISMATCH
FILE_CORRUPTED
```

## 7. 资源消耗限制

默认配置建议：

| 资源 | 默认限制 | 环境变量 |
|---|---:|---|
| 单文件大小 | 50 MB | `D_MODULE_MAX_FILE_SIZE_BYTES` |
| PDF 页数 | 100 | `D_MODULE_MAX_PDF_PAGES` |
| PDF 单页渲染像素 | 50 MP | `D_MODULE_MAX_PDF_RENDER_PIXELS` |
| 图片总像素 | 50 MP | `D_MODULE_MAX_IMAGE_PIXELS` |
| 图片宽/高 | 20000 px | `D_MODULE_MAX_IMAGE_DIMENSION` |
| DOCX 解压总大小 | 200 MB | `D_MODULE_MAX_ARCHIVE_UNPACKED_BYTES` |
| DOCX 内部文件数 | 1000 | `D_MODULE_MAX_ARCHIVE_ENTRIES` |
| 单文件处理时间 | 120 秒 | `D_MODULE_PROCESS_TIMEOUT_SECONDS` |
| 临时文件保留时间 | 1 小时 | `D_MODULE_QUARANTINE_TTL_SECONDS` |

### 7.1 图片像素检查

解码前先读取尺寸元数据，解码后再次确认：

```text
width * height <= 50,000,000
width <= 20,000
height <= 20,000
```

拒绝 `width * height` 过大的图片，避免压缩文件很小但解码后占用数 GB 内存。

### 7.2 PDF 检查

打开 PDF 后立即检查：

- 页数上限
- 页面宽高上限
- 单页渲染像素上限
- 嵌入文件数量
- JavaScript 和外部 URI
- 单页渲染超时

PDF 文本提取和图像渲染都必须使用超时控制。

### 7.3 DOCX 检查

DOCX 按 ZIP 处理，但不直接调用普通解压到正式目录：

1. 遍历 ZIP 条目
2. 拒绝绝对路径和 `..` 路径
3. 检查条目数量和压缩后大小
4. 拒绝宏文件，例如 `vbaProject.bin`
5. 拒绝外部关系和外部 URL
6. 只解压允许的 XML、图片和关系文件
7. 输出到隔离目录

## 8. 恶意内容扫描

### 8.1 扫描接口

定义内部扫描接口：

```text
scan_file(quarantine_path, file_manifest) -> ScanResult
```

返回：

```json
{
  "status": "clean",
  "engine": "clamav",
  "signature": null,
  "scanned_at": "2026-07-18T10:00:00Z"
}
```

状态：

- `clean`：允许继续
- `infected`：拒绝并删除文件
- `error`：不允许继续处理，等待重试或人工处理
- `skipped`：仅允许开发环境使用，生产环境禁止默认跳过

### 8.2 ClamAV 部署

生产环境建议：

- ClamAV 作为独立服务运行
- OCR API 通过受限网络调用扫描服务
- 病毒库定期更新
- 扫描超时后进入 `SCAN_ERROR`
- 扫描结果写入审计日志

病毒扫描不能替代文件格式校验；两者必须同时通过。

## 9. 标准化与放行

### 9.1 图片标准化

通过安全校验后，对图片执行：

- 使用 Pillow/OpenCV 重新解码并编码
- 移除 EXIF、ICC 和非必要元数据
- 限制颜色模式为灰度或 RGB
- 保持原始文件哈希和标准化文件哈希分别记录

原始文件用于证据留存，标准化文件用于 OCR。

### 9.2 PDF/DOCX 标准化

- PDF 使用受限渲染器生成页面图像
- DOCX 先提取文本和图片，再按当前模块的混合解析流程处理
- 不向 OCR 传递外部链接、宏和嵌入对象
- 转换输出只写入正式处理目录

### 9.3 原子放行

放行过程：

```text
quarantine/payload.part
    -> 安全检查全部通过
    -> 写入 accepted/{file_id}/original.ext.tmp
    -> fsync/关闭文件
    -> 原子 rename 为 original.ext
    -> 写入 manifest.json
    -> 状态更新为 ACCEPTED
```

任何一步失败都删除 `.tmp` 文件，不能生成半成品正式文件。

## 10. 文件清单和完整性校验

每个已放行文件保存 `manifest.json`：

```json
{
  "file_id": "file_01H...",
  "submission_id": "sub_001",
  "exam_id": "mayuan_1516",
  "purpose": "answer_sheet",
  "origin_name": "student_001.pdf",
  "extension": "pdf",
  "detected_mime": "application/pdf",
  "size": 1839201,
  "sha256": "...",
  "normalized_sha256": "...",
  "security_status": "clean",
  "scanner": "clamav",
  "created_at": "2026-07-18T10:00:00Z"
}
```

处理任务只接受 `manifest.security_status == "clean"` 且文件哈希存在的文件。

## 11. 与现有接口的兼容规则

成功处理结果继续使用：

```text
schema_version = exam-document.v1
```

不改变以下已有输入：

- `file`
- `document_role`
- `exam_id`
- `submission_id`

不改变以下成功输出：

- 试题卷的 `questions[]`
- 答题卡的 `answers[]`
- `template_id`
- `template_version`
- `question_id`
- `question_no`

安全失败使用统一错误结构：

```json
{
  "detail": {
    "error_code": "FILE_SIGNATURE_MISMATCH",
    "message": "文件扩展名与真实内容不一致",
    "retryable": false,
    "file_id": null
  }
}
```

建议状态码：

| 状态码 | 场景 |
|---:|---|
| 400 | 请求缺少文件或文件名非法 |
| 413 | 文件或请求体超过大小限制 |
| 415 | 文件类型或 MIME 不支持 |
| 422 | 文件损坏、签名不匹配、页数/像素超限 |
| 429 | 上传或处理频率超过限制 |
| 503 | 病毒扫描服务或安全检查服务不可用 |

安全错误不得返回服务器绝对路径、临时文件路径、解析器堆栈或内部命令。

## 12. OCR 工作流闭环

### 12.1 试题卷

```text
上传试题卷
 -> secure_upload(question_paper)
 -> 文件安全放行
 -> create_file_task
 -> 页面渲染/OCR
 -> 自由版式题目切分
 -> questions[]
 -> exam_id 存储题目目录
 -> 结果持久化和审计
```

### 12.2 空白答题卡

```text
上传空白答题卡
 -> secure_upload(answer_sheet_template)
 -> 安全放行
 -> 自动候选区域识别
 -> 教师人工复核
 -> question_id 校验
 -> 模板发布
 -> exam_id + template_id + version 绑定
 -> 审计记录
```

### 12.3 学生答题卡

```text
上传学生答题卡
 -> secure_upload(answer_sheet)
 -> 安全放行
 -> 根据 exam_id 获取已发布模板
 -> 页面配准
 -> 固定区域裁剪
 -> 空白检测/手写 OCR
 -> question_id 关联
 -> answers[]
 -> 结果持久化
 -> 进入批改模块
```

安全校验失败时，链路在进入 OCR 前终止；OCR 失败时，不删除原始安全文件；结果持久化失败时，不生成“处理成功”状态。

## 13. OCR 文本安全边界

学生手写内容是不可信业务数据，必须与系统控制信息隔离：

- OCR 文本不能拼接到系统指令中执行
- OCR 文本不能修改 `exam_id`、`template_id` 或 `question_id`
- OCR 文本中的控制字符、超长内容和异常 Unicode 需要清洗或转义
- 批改模块接收结构化答案字段，不接收可执行模板或指令
- 保存原始 OCR 文本和清洗后文本的关联关系

## 14. 审计和清理闭环

### 14.1 审计事件

至少记录：

```text
UPLOAD_STARTED
UPLOAD_QUARANTINED
SIGNATURE_CHECKED
FORMAT_CHECKED
RESOURCE_CHECKED
MALWARE_SCANNED
FILE_ACCEPTED
FILE_REJECTED
PROCESSING_STARTED
PROCESSING_COMPLETED
PROCESSING_FAILED
QUARANTINE_DELETED
```

每条事件包含：

- `event_id`
- `file_id`
- `submission_id`
- `exam_id`
- 状态转移前后状态
- 时间
- 操作者或服务身份
- 原因和错误码
- 文件 SHA-256

### 14.2 清理任务

定时任务每 10 分钟执行：

1. 删除超过 TTL 的 `quarantine` 临时文件
2. 删除失败任务遗留的 `.tmp` 文件
3. 检查无 manifest 的正式文件
4. 检查扫描超时任务
5. 记录清理结果

正式文件不能因为临时清理任务被删除；正式文件删除必须遵循考试数据保留策略。

## 15. 访问和部署安全

- API 生产环境必须启用 HTTPS
- 上传接口要求登录和角色权限
- 教师只能上传或查看授权考试的文件
- 学生提交只能使用已绑定考试，不能指定模板路径
- OCR Worker 使用非管理员账户运行
- 转换和扫描容器默认无网络访问
- 容器限制 CPU、内存、进程数和临时磁盘
- `storage` 不直接暴露静态访问
- 模型文件和依赖使用固定版本和 SHA-256 校验

## 17. 测试方案

### 17.1 正常样本

- 正常 JPG、PNG、PDF、DOCX
- 中文文件名
- 大小接近上限的合法文件
- 多页试题卷
- 空白答题卡模板
- 学生手写答题卡

### 17.2 类型伪装

- `.pdf` 实际为 JPG
- `.jpg` 实际为 ZIP
- `.docx` 实际为可执行文件
- 客户端 MIME 与文件内容不一致
- 无扩展名文件

### 17.3 资源攻击

- 超过 50 MB 文件
- 超大分辨率图片
- PDF 超过最大页数
- DOCX 高压缩比 ZIP
- DOCX 内部文件数量超限
- 深层 ZIP 路径和 `../` 路径
- 重复上传和并发上传

### 17.4 失败闭环

每个失败用例必须验证：

- 返回正确状态码和错误码
- 不进入 OCR
- 不创建 `FileTask`
- 不写入最终结果
- 隔离区文件被清理
- 审计事件存在

### 17.5 接口回归

- 成功响应仍通过 `exam-document.v1` Schema
- `question_paper` 输出不变
- `answer_sheet` 输出不变
- 模板上传和 `/api/process` 使用同一安全校验
- 相同 `submission_id` 的幂等逻辑不被改变

## 18. 验收标准

方案完成后必须满足：

1. 所有文件入口都经过统一安全上传服务。
2. 未通过真实性校验的文件不会进入 OCR 或文档解析。
3. 超大文件、超大图片、超页数 PDF 和解压炸弹会被拒绝。
4. DOCX 不会将绝对路径或 `../` 文件写入正式目录。
5. 病毒扫描异常时不会放行文件。
6. 任何失败都不会留下可被后续任务读取的半成品文件。
7. 文件安全状态、哈希和处理结果可以关联查询。
8. 成功识别接口输入输出与 `exam-document.v1` 保持兼容。
9. 27 项现有回归测试继续通过，并新增安全测试全部通过。
10. 在无 OpenVINO 或安全扫描服务时，健康检查和错误码能准确反映不可用状态。

## 19. 结论

本方案的核心不是在现有校验函数上增加几个判断，而是把上传文件放入独立的安全生命周期：

```text
接收受限
 -> 隔离保存
 -> 内容确认
 -> 资源限制
 -> 恶意扫描
 -> 原子放行
 -> 受限处理
 -> 结果持久化
 -> 审计追踪
 -> 按策略清理
```

这样可以同时保护上传接口、文档解析器、OCR 引擎、模板生成流程和后续 AI 批改工作流，并且不改变当前识别模块对外的成功响应格式。
